# 任務生成原則

## 核心原則

### 1. 自然語言描述
- 任務描述應該回答"做什麼",而非"如何做"
- 使用能力導向語言(capability-driven),聚焦業務價值
- 避免直接暴露技術實作細節(如類別名稱、函數簽名)

**範例**:
- ✅ 好: "實作圖像預處理功能,移除紅色浮水印並增強文字清晰度"
- ❌ 差: "實作 TranscriptPreprocessor 類別的 remove_red_watermark() 方法"

### 2. 任務大小
- 每個子任務應該在 **1-3 小時**內完成
- 主要任務(Major Task)是邏輯分組,不是可執行單元
- 如果主要任務只有 1 個子任務,將其提升為主要任務(避免不必要的層級)

### 3. 完整需求覆蓋
- 所有需求(requirements.md)必須映射到至少一個任務
- 在每個任務中標註對應的需求 ID(如: 對應需求: 1, 2, 3)
- 非功能性需求(效能、安全性、可用性)也必須有對應任務

### 4. 任務層級
- **最多 2 層**: 主要任務(Major Task) + 子任務(Sub-task)
- 主要任務編號: 1, 2, 3, 4...
- 子任務編號: 1.1, 1.2, 2.1, 2.2...
- 禁止更深的嵌套(如 1.1.1)

### 5. 順序編號
- 主要任務編號遞增,**絕不重複**
- 即使需要回頭修改舊任務,也應該新增新的主要任務編號
- 子任務編號在主要任務內遞增

### 6. 測試優先
- 每個功能任務都應該有對應的測試任務
- 測試任務應該緊跟在功能任務之後
- 測試應該涵蓋:單元測試、整合測試、驗收測試

### 7. 任務整合
- 每個任務都必須與系統整合(避免孤立的工作)
- 明確指出任務的輸入、輸出、依賴關係
- 確保任務完成後系統處於可運行狀態

### 8. 依賴管理
- 明確標註任務依賴關係
- 依賴格式: "依賴: [任務 X.Y]"
- 避免循環依賴

## 任務模板結構

```markdown
## Major Task X: [能力描述]

**目標**: [1-2 句話說明這組任務的業務目標]

### Sub-task X.1: [具體可執行任務]

**描述**: [詳細說明要做什麼,為何重要]

**驗收標準**:
- [ ] 標準 1
- [ ] 標準 2
- [ ] 標準 3

**對應需求**: [需求 ID 列表]
**依賴**: [依賴任務 ID,如無則標註"無"]
**預估時間**: [1-3 小時]
```

## 任務生成檢查清單

生成任務後,必須驗證:
- [ ] 所有需求都有對應任務
- [ ] 每個子任務 1-3 小時
- [ ] 主要任務編號遞增無重複
- [ ] 最多 2 層任務層級
- [ ] 測試任務完整
- [ ] 依賴關係明確
- [ ] 使用自然語言描述
- [ ] 每個任務都整合到系統

## 特殊任務類型

### 研究任務
當需求或設計有不確定性時,建立研究任務:
- 明確研究問題
- 定義成功標準
- 設定時間上限

### 重構任務
改善現有程式碼品質:
- 明確重構目標(效能、可讀性、可測試性)
- 確保向後相容性
- 包含回歸測試

### 部署任務
將功能部署到環境:
- 包含資料庫遷移
- 包含配置更新
- 包含回滾計劃
- 包含煙霧測試

## 常見錯誤

### ❌ 錯誤 1: 任務過大
```
任務 1.1: 實作完整的 OCR 增強系統 (預估: 40 小時)
```

**修正**: 拆分為多個 1-3 小時的子任務

### ❌ 錯誤 2: 技術細節暴露
```
任務 1.1: 實作 TranscriptPreprocessor.__init__() 方法
```

**修正**: "實作圖像預處理器的初始化與配置管理"

### ❌ 錯誤 3: 缺少測試
```
任務 1: 實作預處理模組
  任務 1.1: 實作浮水印移除
  任務 1.2: 實作二值化
(沒有測試任務)
```

**修正**: 新增 "任務 1.3: 預處理模組單元測試"

### ❌ 錯誤 4: 任務不整合
```
任務 1.1: 建立 HSV 色彩轉換函數
(獨立函數,沒有整合到系統)
```

**修正**: "實作浮水印移除功能並整合到預處理流程"

### ❌ 錯誤 5: 需求未覆蓋
```
需求 6: PDF 多頁處理優化
(沒有對應的任務)
```

**修正**: 新增 "任務 X: 實作 PDF 多頁處理優化"
