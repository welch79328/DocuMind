# 設計審查指南

## 審查目標

確保技術設計文件在進入實施階段前,具備以下特質:
1. **完整性**: 涵蓋所有需求,無遺漏
2. **可行性**: 技術方案可實施,風險可控
3. **品質**: 架構合理,符合設計原則
4. **可測試性**: 有明確的驗證方法

## 審查流程

### 1. 分析階段
**檢查項目**:
- [ ] 設計文件結構完整
- [ ] 所有需求都有對應的設計元件
- [ ] 架構圖清晰,邊界明確
- [ ] 型別定義完整,使用 type hints
- [ ] 資料流程圖完整
- [ ] 測試策略定義明確
- [ ] 風險識別完整

### 2. 關鍵問題識別
**限制**: 最多 3 個最重要的問題
**嚴重程度**:
- **Critical**: 阻礙實施,必須解決
- **Major**: 影響品質或效能,強烈建議解決
- **Minor**: 改善建議,可接受現況

**問題格式**:
```
問題 X: [問題標題]
- 嚴重程度: Critical/Major/Minor
- 描述: [詳細說明問題]
- 影響: [對專案的影響]
- 建議: [具體可行的解決方案]
```

### 3. 優點識別
**目的**: 平衡審查,認可設計優點
**數量**: 1-2 個主要優點

### 4. GO/NO-GO 決策
**GO (批准設計)**:
- 無 Critical 問題
- Major 問題 ≤ 2 個且有緩解方案
- 設計可實施,風險可控

**NO-GO (需要修訂)**:
- 存在 Critical 問題
- Major 問題 > 3 個
- 缺少關鍵元件或流程

## 審查檢查清單

### 架構設計
- [ ] 模組劃分合理,職責單一
- [ ] 模組間依賴清晰,避免循環依賴
- [ ] 使用合適的設計模式
- [ ] 擴展性與可維護性考慮

### 型別安全
- [ ] 所有公開介面有型別定義
- [ ] 使用 TypedDict 或 dataclass
- [ ] 避免使用 `any` 型別
- [ ] 邊界處有型別驗證

### 資料流程
- [ ] 資料流程圖完整
- [ ] 錯誤處理路徑明確
- [ ] 降級機制定義清楚
- [ ] 輸入輸出格式明確

### 效能與可靠性
- [ ] 效能目標明確且可測量
- [ ] 錯誤處理策略完整
- [ ] 超時與重試機制
- [ ] 資源用量(記憶體/CPU)考慮

### 測試策略
- [ ] 單元測試範圍定義
- [ ] 整合測試場景定義
- [ ] 驗收測試標準明確
- [ ] 測試覆蓋率目標合理(80%+)

### 部署與維護
- [ ] 部署步驟清晰
- [ ] 配置管理完整
- [ ] 回滾計劃定義
- [ ] 監控指標明確

## 輸出格式

```markdown
# 設計審查報告 - [Feature Name]

## 審查摘要
[2-3 句話總結設計品質與就緒程度]

## 關鍵問題

### 問題 1: [標題]
- **嚴重程度**: Critical/Major/Minor
- **描述**: [詳細說明]
- **影響**: [對專案的影響]
- **建議**: [解決方案]

### 問題 2: [標題]
...

### 問題 3: [標題]
...

## 設計優點

1. **[優點標題]**: [說明]
2. **[優點標題]**: [說明]

## 最終評估

**決策**: ✅ GO / ❌ NO-GO

**理由**: [決策理由]

**下一步**:
- [具體行動項目]
```

## 審查原則

1. **客觀中立**: 基於事實與標準,避免主觀偏見
2. **聚焦關鍵**: 最多 3 個問題,避免過度挑剔
3. **可行建議**: 所有建議都必須可實施
4. **平衡評估**: 認可優點,指出問題
5. **質量導向**: 接受合理的權衡,不追求完美
