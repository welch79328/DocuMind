# 並行任務分析規則

## 目的

識別可並行執行的任務,在任務描述中標註 `(P)` 標記,幫助開發者優化實施順序。

## 並行性判斷標準

### ✅ 可並行 (標註 (P))

任務滿足以下**所有條件**時可並行:

1. **無依賴關係**: 任務之間沒有直接的輸入/輸出依賴
2. **獨立模組**: 任務操作不同的程式碼模組或檔案
3. **無共享資源**: 不修改相同的資料庫表、配置檔案、或共享狀態
4. **可獨立測試**: 任務完成後可獨立驗證,不需要其他任務完成

**範例**:
- ✅ "任務 1.1: 實作浮水印移除功能 (P)"
- ✅ "任務 1.2: 實作二值化功能 (P)"
  → 兩者操作不同的圖像處理步驟,無依賴

### ❌ 不可並行 (無標記)

任務滿足以下**任一條件**時不可並行:

1. **有依賴關係**: 任務 B 需要任務 A 的輸出
2. **修改相同模組**: 兩個任務修改同一個檔案的不同部分
3. **共享資源衝突**: 兩個任務修改同一個資料庫表
4. **順序邏輯**: 任務必須按特定順序執行(如: 建立 → 測試 → 部署)

**範例**:
- ❌ "任務 1.1: 定義 OCR 結果資料模型"
- ❌ "任務 1.2: 實作 OCR 結果儲存邏輯"
  → 任務 1.2 依賴任務 1.1 的模型定義

## 標註格式

### 主要任務 (Major Task)
- 主要任務本身不執行,**不標註 (P)**
- 僅標註可並行的子任務

### 子任務 (Sub-task)
- 可並行: `### Sub-task X.Y: [描述] (P)`
- 不可並行: `### Sub-task X.Y: [描述]`

## 分析流程

### Step 1: 識別依賴關係
- 檢查任務描述中的"依賴"欄位
- 分析任務的輸入/輸出
- 識別資料流向

### Step 2: 檢查模組衝突
- 檢查任務操作的檔案/模組
- 識別是否修改相同的程式碼區域
- 確認資料庫 schema 變更是否衝突

### Step 3: 驗證獨立性
- 確認任務可以獨立完成
- 確認任務可以獨立測試
- 確認任務失敗不影響其他並行任務

### Step 4: 標註並行任務
- 滿足所有並行條件的任務標註 `(P)`
- 不確定時保守處理(不標註)

## 範例場景

### 場景 1: 獨立模組開發 ✅

```markdown
## Major Task 1: 核心模組開發

### Sub-task 1.1: 實作文件分類器模組 (P)
**依賴**: 無
**操作檔案**: `document_classifier.py`

### Sub-task 1.2: 實作圖像預處理模組 (P)
**依賴**: 無
**操作檔案**: `preprocessor.py`

### Sub-task 1.3: 實作引擎管理器模組 (P)
**依賴**: 無
**操作檔案**: `engine_manager.py`
```

**理由**: 三個任務操作不同的模組,無依賴,可並行開發。

---

### 場景 2: 有依賴關係 ❌

```markdown
## Major Task 2: 資料模型與儲存

### Sub-task 2.1: 定義 OCR 結果資料模型
**依賴**: 無
**操作檔案**: `models/ocr_result.py`

### Sub-task 2.2: 實作資料庫遷移腳本
**依賴**: 任務 2.1
**操作檔案**: `alembic/versions/xxx_add_ocr_fields.py`

### Sub-task 2.3: 實作 OCR 結果儲存邏輯
**依賴**: 任務 2.1, 2.2
**操作檔案**: `services/ocr_service.py`
```

**理由**: 任務 2.2 和 2.3 依賴 2.1,必須順序執行,無 (P) 標記。

---

### 場景 3: 部分並行 ✅❌

```markdown
## Major Task 3: 測試開發

### Sub-task 3.1: 預處理模組單元測試 (P)
**依賴**: 任務 1.2 完成
**操作檔案**: `tests/unit/test_preprocessor.py`

### Sub-task 3.2: 引擎管理器單元測試 (P)
**依賴**: 任務 1.3 完成
**操作檔案**: `tests/unit/test_engine_manager.py`

### Sub-task 3.3: 端到端整合測試
**依賴**: 任務 3.1, 3.2 完成
**操作檔案**: `tests/integration/test_pipeline.py`
```

**理由**:
- 任務 3.1 和 3.2 測試不同模組,可並行 → 標註 (P)
- 任務 3.3 依賴前兩者,不可並行 → 無標記

---

### 場景 4: 共享資源衝突 ❌

```markdown
## Major Task 4: API 開發

### Sub-task 4.1: 實作文件上傳 API
**依賴**: 無
**操作檔案**: `api/v1/documents.py`

### Sub-task 4.2: 實作文件處理 API
**依賴**: 無
**操作檔案**: `api/v1/documents.py`
```

**理由**: 兩個任務修改同一個檔案,會產生合併衝突,不可並行。

---

## 特殊情況處理

### 情況 1: 測試任務
- 單元測試通常可並行(測試不同模組)
- 整合測試通常不可並行(需要完整系統)
- 驗收測試必須最後執行

### 情況 2: 配置與部署
- 配置檔案修改通常不可並行(避免衝突)
- 資料庫遷移必須順序執行
- Docker 映像建構可與其他任務並行

### 情況 3: 文件撰寫
- API 文件可與實作並行
- 使用者文件通常在實作完成後撰寫
- 技術文件可在開發過程中並行更新

## 標註檢查清單

生成任務後,驗證:
- [ ] 所有無依賴的獨立模組任務標註 (P)
- [ ] 所有有依賴的任務未標註 (P)
- [ ] 修改相同檔案的任務未標註 (P)
- [ ] 並行任務可獨立完成與測試
- [ ] 不確定時未標註 (P)(保守原則)

## 注意事項

1. **保守原則**: 不確定時不要標註 (P),順序執行更安全
2. **團隊規模**: 並行任務數量應考慮團隊人數
3. **測試優先**: 確保並行任務都有對應的獨立測試
4. **文件同步**: 並行開發時需要更頻繁的溝通與文件同步
