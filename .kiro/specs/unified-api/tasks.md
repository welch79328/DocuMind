# unified-api - 實施任務清單

## 概述

### 實施範圍
將 DocuMind 的多步驟文件處理流程整合為單一 `POST /api/v1/analyze` 端點，包含 S3 上傳、OCR 處理、可選問答、用量追蹤。

### 總體工期
預估 3 個階段，約 16 小時

### 任務統計
- **主要任務**: 5 個
- **子任務**: 12 個
- **總預估時間**: 16 小時
- **覆蓋需求**: 1, 2, 3, 4, 5, 6, 7, 8（共 8 個）

---

## Phase 1: 核心端點開發（P0 需求）

**階段目標**: 完成統一分析端點的核心功能，包含檔案驗證、S3 上傳、OCR 處理、統一回應格式

**交付物**:
- ✅ `POST /api/v1/analyze` 端點可用
- ✅ 檔案上傳至 S3 並回傳 CDN URL
- ✅ OCR 處理結果符合統一回應格式

---

## Major Task 1: 定義統一回應格式與資料型別

**目標**: 建立 API 回應的 Pydantic schema，確保型別安全與自描述性

**優先級**: P0

---

### Sub-task 1.1: 建立 AnalyzeResponse 回應 schema

**描述**:
定義統一分析 API 的回應格式，包含檔案資訊、CDN URL、各頁 OCR 結果、結構化欄位、處理統計等。此為後續所有任務的基礎型別定義。

**驗收標準**:
- [ ] AnalyzeResponse 包含：file_name, file_url, document_type, total_pages, pages, answer, stats
- [ ] 每頁結果包含：ocr_raw（文字+信心度）、rule_postprocessed、llm_postprocessed（可選）、structured_data（可選）
- [ ] ProcessingStats 包含：total_time_ms, total_pages, llm_pages_used, estimated_cost
- [ ] 所有型別使用 Pydantic BaseModel，無 Any 型別

**對應需求**: 2

**依賴**: 無

**預估時間**: 1 小時

---

## Major Task 2: 實作統一分析端點

**目標**: 建立 `/api/v1/analyze` 端點，整合檔案驗證、S3 上傳、OCR 處理的完整流程

**優先級**: P0

---

### Sub-task 2.1: 實作檔案驗證與參數解析

**描述**:
在 API 層實作檔案格式驗證（PDF/JPG/JPEG/PNG）、大小限制（20MB）、文件類型檢查（transcript/contract），回傳繁體中文錯誤訊息。

**驗收標準**:
- [ ] 支援 PDF、JPG、JPEG、PNG 格式，不支援的格式回傳 400
- [ ] 檔案超過 20MB 回傳 413
- [ ] 不支援的文件類型回傳 400 並列出支援的類型
- [ ] 所有錯誤訊息為繁體中文，包含統一的 error_code

**對應需求**: 4, 5, 8

**依賴**: 任務 1.1

**預估時間**: 1.5 小時

---

### Sub-task 2.2: 實作 S3 上傳與路徑分配

**描述**:
將上傳的檔案依文件類型儲存至 S3 對應路徑（`uploads/ocr_transcripts/` 或 `uploads/ocr_contracts/`），設定 public-read 權限，回傳 CDN URL。S3 上傳失敗時降級處理（file_url 為 null，不影響 OCR 結果）。

**驗收標準**:
- [ ] 謄本檔案存放至 `uploads/ocr_transcripts/{uuid}.{ext}`
- [ ] 合約檔案存放至 `uploads/ocr_contracts/{uuid}.{ext}`
- [ ] 回傳完整的 CDN URL
- [ ] S3 上傳失敗時，file_url 為 null，OCR 流程繼續執行

**對應需求**: 3

**依賴**: 任務 1.1

**預估時間**: 1.5 小時

---

### Sub-task 2.3: 實作 OCR 處理流程整合

**描述**:
整合現有的 ProcessorFactory 與 OCR 處理管線，支援 PDF 多頁拆解、逐頁處理、結果組裝。不回傳 original_image（base64），節省回應大小。處理單頁失敗時跳過該頁繼續處理其餘頁面。

**驗收標準**:
- [ ] PDF 自動拆頁並逐頁處理
- [ ] 圖片直接處理為單頁
- [ ] 使用 ProcessorFactory 依文件類型路由到正確的處理器
- [ ] 回應不含 original_image（base64）
- [ ] 部分頁面失敗時，其餘頁面結果正常回傳

**對應需求**: 1, 4

**依賴**: 任務 2.1, 2.2

**預估時間**: 2 小時

---

### Sub-task 2.4: 整合端點並註冊路由

**描述**:
將分析服務整合為完整的 API 端點，註冊路由到 FastAPI app，計算處理統計（時間、LLM 使用量、成本），組裝最終的 AnalyzeResponse。

**驗收標準**:
- [ ] `POST /api/v1/analyze` 可透過 Nginx 正常存取
- [ ] 回應包含完整的 ProcessingStats
- [ ] Swagger 文件自動生成端點說明（繁體中文）

**對應需求**: 1, 2, 8

**依賴**: 任務 2.3

**預估時間**: 1.5 小時

## Major Task 4: 實作可選 AI 問答與文檔完善

**目標**: 整合問答功能至統一端點，完善 API 文檔與錯誤處理

**優先級**: P1

---

### Sub-task 4.1: 整合 AI 問答功能

**描述**:
當使用者提供 question 參數時，將 OCR 結果組合為上下文，呼叫 AI 問答服務回答問題。未提供 question 時 answer 欄位為 null，不呼叫 LLM。問答失敗時 answer 為 null 並附帶錯誤訊息。

**驗收標準**:
- [ ] 帶 question 參數時，回應 answer 欄位有 AI 回答
- [ ] 不帶 question 時，answer 為 null，不呼叫 LLM
- [ ] 問答 LLM 失敗時，answer 為 null，其餘結果正常

**對應需求**: 6

**依賴**: 任務 2.4

**預估時間**: 1.5 小時

---

### Sub-task 4.2: 完善 Swagger 文檔與錯誤格式 (P)

**描述**:
為 analyze 端點撰寫完整的 Swagger 文檔，包含參數說明、回應範例、錯誤碼列表。確保所有錯誤回傳統一格式（含 detail 和 error_code），錯誤訊息為繁體中文。

**驗收標準**:
- [ ] Swagger 文檔包含 analyze 端點完整說明與範例
- [ ] 所有錯誤回應格式為 `{"detail": "...", "error_code": "..."}`
- [ ] 錯誤訊息全部為繁體中文

**對應需求**: 8

**依賴**: 任務 2.4

**預估時間**: 1 小時

---

測試 AI 問答功能的正常與異常流程，驗證 Swagger 文檔正確性與錯誤回應格式。

**驗收標準**:
- [ ] 測試帶 question 參數回傳 AI 回答
- [ ] 測試不帶 question 時 answer 為 null
- [ ] 測試錯誤回應格式正確
- [ ] 所有測試通過

**對應需求**: 6, 8
