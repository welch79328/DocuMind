# unified-api - 需求規格

## 概述

### 功能目標
將 DocuMind 現有的多步驟文件處理流程（上傳 → OCR → AI 分析 → 問答）整合為**單一 API 端點**，讓外部使用者只需一次呼叫即可完成文件智能處理。目標是降低 API 串接複雜度，使 DocuMind 成為可直接對外提供的 API 服務。

### 範圍說明
**包含**：
- 單一 `/api/v1/analyze` 端點，接收文件並回傳完整處理結果
- 支援謄本（transcript）與合約（contract）兩種文件類型
- OCR 辨識 + 規則修正 + LLM 文字校正
- 結構化欄位提取
- 可選的 AI 問答功能（傳入問題即回答）
- 用量統計追蹤（呼叫次數、處理頁數、LLM 成本）

**不包含**：
- 使用者認證與授權（MVP 階段不做）
- 批量文件處理（一次只處理一份文件）
- 非同步任務佇列（同步處理，未來可擴展）
- 計費系統（僅追蹤用量，不實作收費）

### 成功標準
1. 外部使用者僅需一次 API 呼叫即可取得 OCR + 結構化欄位結果
2. API 回應時間 < 30 秒（單頁文件，不含 LLM）
3. 回應格式統一、文檔完整，第三方可依文檔自行串接
4. 既有 OCR 測試頁面功能不受影響

## 需求清單

### 需求 1: 統一分析端點

**業務目標**:
外部使用者需要一個簡單的 API 進行文件智能處理，不應要求使用者了解內部的多步驟流程。

**驗收標準** (使用 EARS 格式):
1. API 服務 SHALL 提供 `POST /api/v1/analyze` 端點，接受文件上傳並回傳處理結果
2. API 服務 SHALL 接受以下參數：檔案（必填）、文件類型（選填，預設 transcript）、是否啟用 LLM（選填，預設 true）、問題（選填）
3. WHEN 使用者上傳檔案時，API 服務 SHALL 依序執行：OCR 辨識 → 規則後處理 → LLM 文字校正（若啟用）→ 結構化欄位提取
4. WHEN 使用者同時提供問題參數時，API 服務 SHALL 基於 OCR 結果回答問題並包含在回應中

**測試場景**:
- [ ] 上傳 PDF 謄本，不帶問題 → 回傳 OCR 文字 + 結構化欄位
- [ ] 上傳 JPG 合約，帶問題「合約金額是多少？」→ 回傳 OCR 文字 + 欄位 + AI 回答
- [ ] 不帶檔案呼叫 → 回傳 400 錯誤
- [ ] 上傳不支援的檔案格式 → 回傳 400 錯誤

**優先級**: P0

---

### 需求 2: 統一回應格式

**業務目標**:
API 回應必須結構清晰、自描述，讓第三方開發者無需額外溝通即可理解回傳資料。

**驗收標準** (使用 EARS 格式):
1. API 服務 SHALL 回傳統一的 JSON 格式，包含：檔案資訊、檔案 CDN URL、文件類型、總頁數、各頁 OCR 結果、結構化欄位、AI 回答（若有）
2. API 服務 SHALL 在每頁結果中包含：原始 OCR 文字、OCR 信心度、規則修正後文字、LLM 修正後文字（若有）
3. API 服務 SHALL 在回應中包含處理統計資訊：處理時間、LLM 使用頁數、預估成本
4. IF 處理過程中部分頁面失敗，THEN API 服務 SHALL 回傳成功頁面的結果並標記失敗頁面的錯誤資訊

**測試場景**:
- [ ] 驗證回應 JSON 結構包含所有必要欄位
- [ ] 驗證多頁 PDF 回傳所有頁面結果
- [ ] 驗證部分頁面處理失敗時仍回傳其餘結果

**優先級**: P0

---

### 需求 3: 檔案上傳至 S3

**業務目標**:
上傳的文件需要持久化儲存至 AWS S3，並透過 CloudFront CDN 提供存取，與現有系統的檔案儲存規則保持一致。

**驗收標準** (使用 EARS 格式):
1. WHEN 使用者上傳檔案時，API 服務 SHALL 將原始檔案儲存至 S3
2. API 服務 SHALL 依文件類型使用對應的 S3 路徑：`uploads/ocr_transcripts/{uuid}.{ext}`（謄本）、`uploads/ocr_contracts/{uuid}.{ext}`（合約）
3. API 服務 SHALL 在回應中包含檔案的 CDN URL，格式為 `https://{cdn_domain}/uploads/ocr_{type}/{uuid}.{ext}`
4. API 服務 SHALL 上傳時設定 `public-read` 權限，使檔案可透過 CDN 直接存取
5. IF S3 上傳失敗，THEN API 服務 SHALL 回傳 500 錯誤並說明儲存失敗原因，OCR 處理結果仍正常回傳

**測試場景**:
- [ ] 上傳謄本 PDF → 檔案存在 `uploads/ocr_transcripts/` 路徑下
- [ ] 上傳合約 JPG → 檔案存在 `uploads/ocr_contracts/` 路徑下
- [ ] 回應包含有效的 CDN URL
- [ ] CDN URL 可直接存取該檔案

**優先級**: P0

---

### 需求 4: 文件類型支援

**業務目標**:
不同文件類型需要不同的處理策略與欄位提取邏輯，API 必須正確路由到對應的處理器。

**驗收標準** (使用 EARS 格式):
1. API 服務 SHALL 支援以下文件類型參數：`transcript`（謄本）、`contract`（合約）
2. WHEN 文件類型為 `transcript` 時，API 服務 SHALL 使用謄本處理器進行 OCR 與後處理
3. WHEN 文件類型為 `contract` 時，API 服務 SHALL 使用合約處理器進行 OCR 與後處理，並提取合約結構化欄位
4. IF 傳入不支援的文件類型，THEN API 服務 SHALL 回傳 400 錯誤並列出支援的類型

**測試場景**:
- [ ] document_type=transcript → 使用謄本處理器
- [ ] document_type=contract → 使用合約處理器，回傳結構化欄位
- [ ] document_type=invoice → 回傳 400 錯誤

**優先級**: P0

---

### 需求 5: 檔案驗證

**業務目標**:
保護系統不受無效或惡意檔案影響，提供明確的錯誤訊息引導使用者修正。

**驗收標準** (使用 EARS 格式):
1. API 服務 SHALL 支援以下檔案格式：PDF、JPG、JPEG、PNG
2. API 服務 SHALL 限制單次上傳檔案大小不超過 20MB
3. IF 檔案格式不支援，THEN API 服務 SHALL 回傳 400 錯誤，訊息包含支援的格式清單
4. IF 檔案大小超過限制，THEN API 服務 SHALL 回傳 413 錯誤，訊息包含大小限制

**測試場景**:
- [ ] 上傳 PDF → 處理成功
- [ ] 上傳 PNG → 處理成功
- [ ] 上傳 .docx → 回傳 400 錯誤
- [ ] 上傳 25MB 檔案 → 回傳 413 錯誤

**優先級**: P0

---

### 需求 6: 可選 AI 問答

**業務目標**:
使用者可能只需要 OCR + 欄位提取，也可能想直接問問題。API 應支援兩種使用情境。

**驗收標準** (使用 EARS 格式):
1. WHERE 使用者提供 `question` 參數，API 服務 SHALL 基於 OCR 結果使用 LLM 回答問題
2. WHERE 使用者未提供 `question` 參數，API 服務 SHALL 只回傳 OCR 結果與結構化欄位，不呼叫問答 LLM
3. API 服務 SHALL 在回應中的 `answer` 欄位回傳 AI 回答，未提供問題時該欄位為 null

**測試場景**:
- [ ] 不帶 question → 回應 answer 欄位為 null
- [ ] 帶 question=「所有權人是誰？」→ 回應 answer 欄位有內容
- [ ] 帶 question 但 LLM 服務不可用 → 回傳錯誤訊息

**優先級**: P1

---

### 需求 7: 用量統計追蹤

**業務目標**:
營運方需要掌握 API 使用狀況，為未來計費做準備。

**驗收標準** (使用 EARS 格式):
1. API 服務 SHALL 記錄每次呼叫的：時間戳記、文件類型、處理頁數、是否使用 LLM、LLM 預估成本
2. API 服務 SHALL 提供 `GET /api/v1/usage` 端點查詢用量統計
3. WHEN 查詢用量時，API 服務 SHALL 回傳：總呼叫次數、總處理頁數、總 LLM 成本

**測試場景**:
- [ ] 呼叫 analyze 後，usage 記錄增加
- [ ] 查詢 usage 回傳正確的累計數據
- [ ] 無任何呼叫時，usage 回傳零值

**優先級**: P2

---

### 需求 8: API 文件與錯誤處理

**業務目標**:
第三方開發者需要完整的 API 文件與一致的錯誤格式才能順利串接。

**驗收標準** (使用 EARS 格式):
1. API 服務 SHALL 在 Swagger 文件中提供 `/api/v1/analyze` 的完整說明，包含參數、回應範例、錯誤碼
2. API 服務 SHALL 對所有錯誤回傳統一格式：`{"detail": "錯誤訊息", "error_code": "錯誤代碼"}`
3. API 服務 SHALL 使用繁體中文撰寫所有錯誤訊息
4. IF 處理過程發生未預期錯誤，THEN API 服務 SHALL 回傳 500 錯誤，不洩漏內部實作細節

**測試場景**:
- [ ] Swagger 文件包含 analyze 端點完整說明
- [ ] 所有錯誤回應符合統一格式
- [ ] 錯誤訊息為繁體中文

**優先級**: P1

## 非功能性需求

### 效能需求
- API 服務 SHALL 在 30 秒內完成單頁文件的 OCR 處理（不含 LLM）
- API 服務 SHALL 在 60 秒內完成單頁文件的完整處理（含 LLM 校正）

### 可用性需求
- API 服務 SHALL 透過 Nginx 反向代理對外提供服務（port 8085）
- API 服務 SHALL 在 Docker 容器環境中正常運作

### 安全性需求
- API 服務 SHALL 驗證上傳檔案的 MIME 類型
- API 服務 SHALL 限制請求 body 大小（透過 Nginx client_max_body_size）

---

## 限制與假設

### 技術限制
- 同步處理模式，大文件（>10 頁）處理時間可能較長
- LLM 校正不傳圖片（避免 GPT-4o PII 過濾），僅做文字校正
- 依賴外部 OpenAI API，需要有效的 API Key

### 業務假設
- MVP 階段不需要使用者認證
- 文件類型由呼叫方明確指定，不做自動分類
- 單次只處理一份文件

### 下游影響
- 現有 `/api/v1/ocr/test` 端點保留，供內部測試使用
- 現有 `/api/v1/documents/*` 與 `/api/v1/chat/*` 端點保留，供內部使用
- 前端 API 文件連結需指向新的 Swagger 頁面
