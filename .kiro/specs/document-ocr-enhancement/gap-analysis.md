# document-ocr-enhancement - 實施差距分析報告

## 執行摘要

### 分析範圍
本分析評估 DocuMind 現有 OCR 架構與「政府建物土地謄本 OCR 優化」需求之間的實施差距,涵蓋圖像預處理、多引擎融合、後處理校正、品質評估、專用欄位提取等六大需求領域。

### 主要發現
1. **基礎架構完善**: 現有系統已具備完整的 OCR 服務架構,支援 3 種 OCR 引擎(AWS Textract、PaddleOCR、Tesseract),並整合於文件處理流程中
2. **預處理能力不足**: 現有預處理僅支援基本的灰階、調整大小、對比度/銳度增強,缺少針對浮水印、印章、適應性二值化等專用處理
3. **單引擎模式**: 目前僅支援單一 OCR 引擎選擇,缺少多引擎並行與融合機制
4. **缺少後處理**: 無 OCR 結果校正、格式化、錯別字修正等後處理邏輯
5. **無品質保證機制**: 缺少品質評估、自動重試、信心度評分等品質控制功能
6. **測試覆蓋率為零**: 專案目前無任何單元測試或整合測試

### 建議策略
**推薦方案 C(混合方案)**: 建立新的增強模組(`ocr_enhanced/`)保持向後相容性,同時擴展現有 `ocr_service.py` 增加入口函數,逐步遷移至新架構。此方案平衡了開發效率、程式碼品質與系統穩定性。

## 差距詳細分析

### 需求 1: 政府謄本專用圖像預處理 (P0)

#### 差距描述
現有預處理功能(`ocr_service.py` 第 95-119 行)僅包含:
- 灰階轉換
- 簡單調整大小(基於固定閾值)
- 對比度增強(1.3x)
- 銳度增強(1.5x)

**缺少關鍵功能**:
1. **紅色浮水印檢測與移除**: 政府謄本的紅色「已列印」浮水印嚴重干擾黑色文字辨識
2. **紅色印章處理**: 公章與文字重疊時的分離技術
3. **適應性二值化**: Otsu、Sauvola 等自適應閾值二值化方法
4. **形態學去噪**: 開運算、閉運算移除椒鹽雜訊
5. **文件類型檢測**: 自動識別謄本文件並套用專用預處理流程

#### 技術挑戰
- **浮水印移除精度**: 需平衡浮水印移除與文字保留,避免過度處理
- **顏色空間轉換**: HSV/LAB 色彩空間分離紅色通道
- **參數調優**: 不同掃描品質需要不同的二值化閾值

#### 風險評估
- **風險等級**: 中等
- **主要風險**: 過度預處理可能降低高品質文件的辨識率
- **緩解措施**: 實作降級機制,比較預處理前後的 OCR 信心度

---

### 需求 2: 智能 OCR 引擎選擇與融合 (P0)

#### 差距描述
現有架構僅支援單一引擎模式(透過 `settings.OCR_SERVICE` 配置),執行流程為:
1. 讀取配置值(`textract`/`paddleocr`/`pytesseract`)
2. 調用對應函數
3. 回傳單一引擎結果

**缺少關鍵功能**:
1. **並行處理**: 同時運行多個引擎(需 `asyncio.gather()`)
2. **信心度標準化**: 統一不同引擎的信心度格式(Textract 無信心度、PaddleOCR 逐行信心度、Tesseract 需額外配置)
3. **結果融合算法**: 基於信心度的投票、加權平均、編輯距離比對
4. **引擎選擇策略**: 根據文件類型、語言、歷史表現自動選擇引擎

#### 技術挑戰
- **信心度不一致**: 三個引擎的信心度格式完全不同,需要標準化層
- **融合算法複雜度**: 字符級對齊與融合需要複雜的文字比對算法
- **效能權衡**: 多引擎並行會增加處理時間與 API 成本(特別是 Textract)

#### 風險評估
- **風險等級**: 高
- **主要風險**: 多引擎並行可能導致處理時間超過 90 秒需求(PDF 3頁)
- **緩解措施**: 實作可配置的引擎組合(如僅 PaddleOCR + Tesseract,不包含 Textract)

---

### 需求 3: OCR 結果後處理與校正 (P0)

#### 差距描述
現有系統將 OCR raw_text 直接傳遞給 AI 服務,無任何預處理步驟。

**缺少關鍵功能**:
1. **繁體中文常見錯誤修正**:
   - 注音符號替換(㆞→地、㆗→中、㆟→人等)
   - 全形/半形轉換
   - 繁簡混淆修正
2. **正則表達式校正**:
   - 地號格式: `\d{4}-\d{4}`
   - 統一編號: `[A-Z]\d{9}` 或 `[A-Z]\d{3}\*+\d`
   - 日期格式: `民國\d{2,3}年\d{1,2}月\d{1,2}日`
3. **重複浮水印移除**:
   - 偵測重複出現的文字模式(如每頁都出現「已列印」、「本謄本列印完畢」)
   - 移除或標記為元數據
4. **結構化解析**:
   - 識別區塊標題(土地標示部、所有權部、建物標示部等)
   - 建立階層結構

#### 技術挑戰
- **錯別字字典維護**: 需要建立並持續更新繁體中文 OCR 常見錯誤字典
- **正則表達式精確度**: 避免誤修正(如將正常文字誤認為格式錯誤)
- **結構化解析複雜度**: 謄本格式變化多端,需要靈活的解析邏輯

#### 風險評估
- **風險等級**: 中等
- **主要風險**: 過度校正可能引入新錯誤
- **緩解措施**: 實作信心度閾值,僅修正高信心度錯誤

---

### 需求 4: OCR 品質評估與自動重試 (P1)

#### 差距描述
現有系統無任何品質評估機制,無論 OCR 結果好壞均直接進入下一步處理。

**缺少關鍵功能**:
1. **品質評分算法**:
   - 平均信心度(需先實作需求 2 的信心度標準化)
   - 字符密度(文字/圖像面積比)
   - 已知欄位匹配度(如地號、統一編號格式匹配)
   - 異常字符比例(亂碼、特殊符號)
2. **重試策略管理**:
   - 參數組合管理(3 種預處理參數 × 3 種引擎 = 9 種組合)
   - 重試決策邏輯(何時重試、使用哪種策略)
   - 最佳結果選擇(比較多次嘗試的結果)
3. **品質報告生成**:
   - 結構化品質指標(JSON 格式)
   - 改善建議(基於失敗原因)
   - 處理歷史追蹤

#### 技術挑戰
- **品質閾值設定**: 需要實驗確定合理的閾值(建議從 60 分開始)
- **重試策略爆炸**: 組合太多會導致處理時間過長
- **成本控制**: 每次重試都會消耗 API 配額(特別是 Textract)

#### 風險評估
- **風險等級**: 低
- **主要風險**: 重試機制可能導致處理時間超標
- **緩解措施**: 限制最多 3 次重試,使用免費引擎(PaddleOCR/Tesseract)進行重試

---

### 需求 5: 建物土地謄本專用欄位辨識 (P1)

#### 差距描述
現有系統使用 AI 服務進行通用欄位抽取(`ai_service.py` 第 41-59 行),依賴 OpenAI 理解文字並提取欄位。此方法:
- ✅ 優點: 靈活、適應不同文件格式
- ❌ 缺點: 成本高、速度慢、準確率不穩定

**缺少關鍵功能**:
1. **正則表達式專用提取器**:
   - 地號: `(?:㆞|地)號[：:]\s*(\d{4}-\d{4})`
   - 面積: `面\s*積[：:]\s*([\d,]+\.?\d*)\s*平方公尺`
   - 統一編號: `統㆒編號[：:]\s*([A-Z]\d{2,3}\*+\d{1,2})`
   - 權狀字號: `(\d{2,3})[北中南]建字第(\d+)號`
   - 日期: `民國(\d{2,3})年(\d{1,2})月(\d{1,2})日`
2. **格式驗證器**:
   - 檢查提取結果是否符合已知格式
   - 標記異常值供人工確認
3. **混合提取策略**:
   - 優先使用正則提取(快速、準確、免費)
   - 失敗時回退到 AI 提取(靈活)

#### 技術挑戰
- **正則表達式健壯性**: 需要處理 OCR 錯誤(如「㆞」誤識為「地」)
- **格式變化**: 不同年份、不同地政事務所的謄本格式可能略有差異
- **邊界情況**: 欄位值缺失、格式異常時的處理

#### 風險評估
- **風險等級**: 低
- **主要風險**: 正則表達式過於嚴格導致匹配失敗
- **緩解措施**: 使用寬鬆的正則表達式,結合格式驗證進行後處理

---

### 需求 6: PDF 多頁文件優化處理 (P1)

#### 差距描述
現有 PDF 處理(`ocr_service.py` 第 80-92 行)已包含:
- ✅ 使用 `pdf2image.convert_from_bytes()` 轉換 PDF
- ✅ 逐頁處理並標記頁碼
- ✅ 合併多頁結果

**缺少關鍵功能**:
1. **DPI 配置驗證**: 現有程式碼未指定 DPI,使用 `pdf2image` 預設值(可能為 200)
2. **轉換品質驗證**: 無檢查轉換後圖像品質
3. **跨頁內容處理**: 無特殊處理跨頁表格或欄位

#### 技術挑戰
- **DPI 權衡**: 更高 DPI(300+)提升品質但增加記憶體用量與處理時間
- **跨頁邏輯**: 判斷哪些內容屬於跨頁需要複雜的版面分析

#### 風險評估
- **風險等級**: 極低
- **主要風險**: DPI 設定不當導致圖像過大或品質不足
- **緩解措施**: 實驗確定最佳 DPI(建議 300),並監控記憶體用量

## 實施方案建議

### 方案 A: 擴展現有 ocr_service.py

#### 描述
直接在現有 `ocr_service.py` 中增加新功能:
- 在 `extract_text_with_tesseract()` 和 `extract_text_with_paddleocr()` 中增強預處理邏輯
- 新增 `extract_text_with_multi_engine()` 函數實作融合
- 新增 `postprocess_ocr_text()` 函數實作後處理
- 修改 `extract_text_from_document()` 增加品質評估與重試

#### 優點
- ✅ 實作速度快,無需大幅重構
- ✅ 向後相容,現有 API 無需修改
- ✅ 單一檔案,維護簡單

#### 缺點
- ❌ ocr_service.py 會變得非常龐大(預估 > 800 行)
- ❌ 職責混亂,違反單一職責原則
- ❌ 測試困難,耦合度高
- ❌ 難以針對特定文件類型(謄本)進行優化

#### 工作量估算
- 預處理增強: 3-4 天
- 多引擎融合: 3-4 天
- 後處理校正: 2-3 天
- 品質評估: 2 天
- 專用欄位提取: 2 天
- PDF 優化: 0.5 天
- **總計: 13-15.5 天**

#### 風險等級
**中等** - 程式碼品質下降,長期維護成本高

### 方案 B: 建立新的 ocr_enhanced/ 模組

#### 描述
建立完整的新模組結構:
```
backend/app/lib/ocr_enhanced/
├── __init__.py
├── preprocessor.py          # 圖像預處理
├── engine_manager.py        # 多引擎管理與融合
├── postprocessor.py         # 後處理與校正
├── quality_assessor.py      # 品質評估
├── field_extractor.py       # 專用欄位提取
├── document_classifier.py   # 文件類型檢測
└── config.py                # 預處理參數配置
```

修改 `ocr_service.py` 增加新入口:
```python
async def extract_text_enhanced(file_url: str, doc_type: str = "transcript") -> dict:
    """Enhanced OCR with preprocessing and quality control"""
    from app.lib.ocr_enhanced import EnhancedOCRPipeline
    return await EnhancedOCRPipeline().process(file_url, doc_type)
```

#### 優點
- ✅ 模組化設計,職責清晰
- ✅ 易於測試,每個元件獨立
- ✅ 可擴展性強,未來支援其他文件類型
- ✅ 程式碼品質高,符合 SOLID 原則
- ✅ 可逐步遷移,不影響現有功能

#### 缺點
- ❌ 實作時間較長
- ❌ 需要更多前期設計
- ❌ 可能過度工程化(對於單一文件類型優化)

#### 工作量估算
- 模組架構設計: 1 天
- 預處理模組: 4-5 天
- 引擎管理模組: 4-5 天
- 後處理模組: 3 天
- 品質評估模組: 2-3 天
- 欄位提取模組: 2-3 天
- 整合與測試: 2-3 天
- **總計: 18-23 天**

#### 風險等級
**低** - 架構清晰,長期維護成本低

### 方案 C: 混合方案 (推薦)

#### 描述
結合方案 A 和 B 的優點:
1. **第一階段**: 在 `ocr_service.py` 中快速實作核心功能
   - 新增 `preprocess_for_transcript()` 函數(包含浮水印移除、二值化)
   - 新增 `postprocess_transcript_text()` 函數(錯別字修正、格式校正)
   - 修改現有引擎函數調用這些新函數
2. **第二階段**: 逐步重構為模組化架構
   - 將預處理邏輯抽取到 `ocr_enhanced/preprocessor.py`
   - 將後處理邏輯抽取到 `ocr_enhanced/postprocessor.py`
   - 保持 `ocr_service.py` 作為統一入口,內部調用新模組

#### 優點
- ✅ 快速交付核心功能(第一階段 10-12 天)
- ✅ 逐步改善程式碼品質,降低風險
- ✅ 向後相容,不影響現有功能
- ✅ 靈活調整,根據實際需求決定第二階段範圍

#### 缺點
- ⚠️ 需要兩階段實施,可能延長總工期
- ⚠️ 第一階段程式碼可能需要後續重構

#### 工作量估算
**第一階段(MVP)**:
- 預處理函數: 3-4 天
- 後處理函數: 2-3 天
- 多引擎並行(簡化版): 2-3 天
- 品質評估(基礎版): 1-2 天
- 專用欄位提取: 2 天
- 整合測試: 1-2 天
- **第一階段總計: 11-16 天**

**第二階段(重構,可選)**:
- 模組化重構: 5-7 天
- 完整測試覆蓋: 3-4 天
- **第二階段總計: 8-11 天**

**整體總計: 19-27 天**(可選擇僅實施第一階段)

#### 風險等級
**低** - 漸進式交付,風險可控

#### 推薦理由
1. **快速見效**: 第一階段即可滿足 P0 需求,提升 OCR 準確率
2. **降低風險**: 逐步重構降低引入 bug 的風險
3. **靈活度高**: 可根據第一階段效果決定是否實施第二階段
4. **成本效益**: 平衡開發效率與程式碼品質

## 技術研究需求

### 1. 浮水印移除技術研究 (高優先級)

#### 研究問題
- 紅色浮水印的最佳移除演算法(色彩空間分離 vs 頻域過濾)
- 浮水印移除對黑色文字的影響評估
- 參數調優(HSV 閾值、形態學操作核大小)

#### 建議方法
- PoC 實驗: 對比 OpenCV 的 HSV 過濾、LAB 色彩空間、頻域過濾(FFT)
- 評估指標: 浮水印移除率、文字保留率、OCR 準確率變化
- 參考資源:
  - OpenCV 官方教學: Color Space Conversions
  - 論文: "Watermark Removal in Document Images" (IEEE)

### 2. 繁體中文 OCR 錯別字字典建立 (中優先級)

#### 研究問題
- 繁體中文 OCR 常見錯誤類型與頻率
- 錯別字字典的資料結構與查詢效率
- 錯別字修正的信心度評估

#### 建議方法
- 資料收集: 對現有謄本文件進行 OCR,人工標註錯誤
- 字典建立: 使用 Trie 樹或 Hash Map 儲存錯別字對照表
- 評估方法: 修正準確率、誤修正率、處理時間
- 參考資源:
  - 開源專案: `python-Levenshtein` (編輯距離計算)
  - 字典資料: 教育部異體字字典、常見錯別字表

### 3. 多引擎結果融合演算法 (中優先級)

#### 研究問題
- 字符級對齊演算法(考慮 OCR 誤差)
- 信心度加權策略
- 融合結果的準確率提升幅度

#### 建議方法
- 演算法比較: 投票演算法、加權平均、編輯距離對齊
- PoC 實驗: 使用相同文件對比不同演算法效果
- 評估指標: 準確率提升、處理時間增加、記憶體用量
- 參考資源:
  - 論文: "Ensemble Methods for OCR" (ACM)
  - 開源專案: `ocr-ensemble` (GitHub)

### 4. 品質評分算法設計 (低優先級)

#### 研究問題
- 品質指標的權重分配
- 品質閾值的實驗確定
- 品質分數與實際 OCR 準確率的相關性

#### 建議方法
- 特徵工程: 信心度、字符密度、格式匹配度、異常字符比例
- 機器學習: 使用歷史資料訓練品質預測模型(可選)
- 閾值調優: A/B 測試確定最佳閾值
- 參考資源:
  - 論文: "OCR Quality Assessment" (Springer)

### 資料模型變更

#### 方案 A: 擴展現有 DocumentOcrResult 模型

**新增欄位**:
```python
class DocumentOcrResult(Base):
    # ... 現有欄位 ...

    # 新增欄位
    preprocessing_params = Column(JSONB, nullable=True)  # 預處理參數
    quality_score = Column(Numeric(5, 2), nullable=True)  # 品質分數(0-100)
    engines_used = Column(String(200), nullable=True)    # 使用的引擎列表
    retry_count = Column(Integer, default=0)              # 重試次數
```

**優點**: 無需資料庫遷移大量欄位,向後相容
**缺點**: 單一表格欄位過多

#### 方案 B: 建立新的 OcrQualityMetrics 表格

**新增表格**:
```python
class OcrQualityMetrics(Base):
    __tablename__ = "ocr_quality_metrics"

    id = Column(UUID, primary_key=True, default=uuid.uuid4)
    ocr_result_id = Column(UUID, ForeignKey("document_ocr_results.id"))
    quality_score = Column(Numeric(5, 2), nullable=False)
    character_density = Column(Numeric(5, 4), nullable=True)
    field_match_rate = Column(Numeric(5, 4), nullable=True)
    anomaly_rate = Column(Numeric(5, 4), nullable=True)
    recommendations = Column(JSONB, nullable=True)
    created_at = Column(DateTime, server_default=func.now())
```

**優點**: 關注點分離,資料模型清晰
**缺點**: 需要額外的表格關聯查詢

**推薦**: 方案 A(擴展現有模型),更簡單且符合需求

4. 整合測試與調優

**交付物**: OCR 準確率提升 15%,滿足成功標準 1-2

#### Phase 2: 多引擎與品質控制(1.5 週)
1. 實作多引擎並行處理(需求 2)
2. 實作品質評估與重試機制(需求 4)
3. 新增品質報告 API

**交付物**: 品質保證機制,降低低品質結果比例

#### Phase 3: 專用欄位提取(1 週)
1. 實作正則表達式欄位提取器(需求 5)
2. 整合到現有 AI 抽取流程(混合策略)

**交付物**: 關鍵欄位準確率 > 95%,滿足成功標準 3

#### Phase 4: 重構與測試(可選,1.5 週)
1. 模組化重構
2. 完整單元測試與整合測試
3. 效能優化

**總工期**: 4.5-6 週(不含 Phase 4 為 3-4 週)

### 成功關鍵因素

1. **早期驗證**: Phase 1 完成後立即使用目標文件(`建物土地謄本-杭州南路一段.pdf`、`建物謄本.jpg`)驗證效果
2. **參數調優**: 預處理參數需要實驗調優,預留 2-3 天調優時間
3. **錯別字字典**: 繁體中文錯別字字典的品質直接影響後處理效果,建議從現有資源收集
4. **測試資料不變**: 嚴格遵守「不修改測試資料」的約束,確保比較基準一致

### 風險緩解

1. **處理時間風險**: 實作超時檢測,超過 80% 時間限制時自動降級為單引擎模式
2. **過度處理風險**: 實作 A/B 比較,保留原始 OCR 結果,若增強後信心度下降則使用原始結果
3. **成本風險**: 優先使用免費引擎(PaddleOCR、Tesseract),僅在必要時使用 Textract
4. **相容性風險**: 保持向後相容,新功能透過配置開關控制,可隨時回退
