# multi-document-type-ocr - 需求規格

## 概述

### 功能目標
擴展現有的 OCR 辨識系統,從單一的謄本文件類型支援,升級為支援多種文件類型(謄本、合約等)的通用 OCR 平台。透過在 `/api/v1/ocr/test` API 中增加文件類型參數,使系統能夠根據不同文件類型應用專門的 OCR 處理策略,並返回符合該文件類型特性的結構化結果。

### 範圍說明

**包含範圍**:
- 在現有 API 中新增文件類型參數 (`document_type`)
- 實作合約文件的專用 OCR 處理流程
- 設計並實作類型特定的回應格式結構
- 利用 `data/contracts/` 中的 11 份合約範例進行開發與測試
- 保持與現有謄本辨識功能的向後兼容性
- 建立可擴展的架構,便於未來新增其他文件類型

**不包含範圍**:
- 建立新的 API 端點(沿用現有 `/api/v1/ocr/test`)
- 修改現有謄本辨識的核心邏輯
- 自動文件類型分類(需由用戶明確指定類型)
- 文件內容語意分析或摘要生成
- 多文件批次處理功能

### 成功標準
1. API 能夠透過 `document_type` 參數正確區分謄本和合約文件
2. 合約文件的 OCR 準確率達到 85% 以上
3. 謄本文件的回應格式維持不變(向後兼容)
4. 合約文件能夠返回結構化的關鍵欄位(合約編號、甲乙方、金額、日期等)
5. 所有 11 份範例合約均能成功辨識並提取關鍵資訊
6. 處理時間不因文件類型判斷而顯著增加(< 5% overhead)

## 需求清單

### 需求 1: 文件類型參數化支援

**業務目標**:
讓使用者能夠透過統一的 API 端點處理不同類型的文件,避免維護多個 API,同時確保系統能針對不同文件類型應用最佳的處理策略。

**驗收標準** (使用 EARS 格式):
1. API SHALL 接受 `document_type` 參數,支援的值包括 `transcript`(謄本) 和 `contract`(合約)
2. WHEN `document_type` 參數未提供時,系統 SHALL 預設為 `transcript` 以保持向後兼容性
3. IF `document_type` 參數值不在支援列表中,THEN 系統 SHALL 返回 400 錯誤並列出支援的文件類型
4. 系統 SHALL 在回應中包含所使用的 `document_type` 值,便於追蹤與除錯
5. 系統 SHALL 記錄每次請求的文件類型,用於使用統計與優化決策

**測試場景**:
- [ ] 場景 1: 不提供 `document_type` 參數上傳謄本,驗證系統預設使用 `transcript` 類型
- [ ] 場景 2: 明確指定 `document_type=contract` 上傳合約,驗證系統使用合約處理流程
- [ ] 場景 3: 提供不支援的 `document_type=invoice`,驗證返回 400 錯誤與明確的錯誤訊息
- [ ] 場景 4: 檢查回應中的 `document_type` 欄位,確認與請求參數一致

**優先級**: P0

---

### 需求 2: 合約文件專用 OCR 處理

**業務目標**:
合約文件具有不同於謄本的特徵(例如:多頁長文、條款編號、表格、簽名欄等),需要針對性的 OCR 處理策略,以提升辨識準確度與關鍵資訊提取的成功率。

**驗收標準** (使用 EARS 格式):
1. 系統 SHALL 為合約文件實作專用的預處理流程
2. WHEN 處理合約文件時,系統 SHALL 針對表格結構進行版面分析,保留表格語意
3. WHEN 合約包含簽名或印章時,系統 SHALL 識別並標記這些區域,避免干擾文字辨識
4. 系統 SHALL 對合約中的條款編號(如「第一條」、「1.1」)進行特別處理,確保結構完整性
5. WHEN 合約文件超過 10 頁時,系統 SHALL 針對每頁生成獨立的辨識結果,並保留頁碼資訊
6. 系統 SHALL 支援合約常見的繁體中文法律用語辨識(如「茲經」、「立合約書人」等)
7. IF 合約預處理失敗,THEN 系統 SHALL 回退到通用 OCR 流程並記錄警告

**測試場景**:
- [ ] 場景 1: 使用 `data/contracts/` 中包含表格的合約,驗證表格內容能正確提取
- [ ] 場景 2: 處理包含簽名欄的合約,驗證簽名不干擾周圍文字辨識
- [ ] 場景 3: 辨識包含「第一條」、「第二條」等條款編號的合約,驗證編號與內容的對應關係
- [ ] 場景 4: 處理超過 10 頁的長合約,驗證每頁結果包含正確的頁碼
- [ ] 場景 5: 使用所有 11 份範例合約測試,驗證成功率達 100%

**優先級**: P0

### 需求 3: 類型特定的回應格式

**業務目標**:
不同文件類型需要返回不同結構的結果,以符合使用者對該文件類型的期望。例如合約需要提取合約編號、甲乙方、金額等結構化欄位,而謄本則需要地號、面積等資訊。

**驗收標準** (使用 EARS 格式):
1. WHEN `document_type=transcript` 時,系統 SHALL 返回與現有格式完全一致的回應結構(向後兼容)
2. WHEN `document_type=contract` 時,系統 SHALL 返回包含以下結構化欄位的回應:
   - `contract_metadata`: 合約元資訊(合約編號、簽訂日期、生效日期)
   - `parties`: 合約雙方資訊(甲方、乙方的名稱、地址、聯絡方式)
   - `financial_terms`: 財務條款(合約金額、付款方式、付款期限)
   - `key_clauses`: 關鍵條款清單(條款編號與內容摘要)
   - `signatures`: 簽名資訊(簽名日期、簽名頁位置)
3. 系統 SHALL 在回應的頂層包含 `document_type` 欄位,明確標示文件類型
4. 系統 SHALL 在回應中包含 `extraction_confidence` 欄位,表示結構化欄位提取的整體信心度
5. WHEN 某個結構化欄位無法提取時,系統 SHALL 返回 `null` 而非省略該欄位
6. 系統 SHALL 在回應中保留完整的原始 OCR 文字(`full_text` 欄位),供使用者參考

**測試場景**:
- [ ] 場景 1: 上傳謄本文件,驗證回應格式與現有系統完全一致
- [ ] 場景 2: 上傳合約文件,驗證回應包含所有必要的結構化欄位
- [ ] 場景 3: 使用缺少合約編號的合約,驗證 `contract_metadata.contract_number` 為 `null`
- [ ] 場景 4: 檢查合約回應中的 `extraction_confidence`,確認為 0-1 之間的數值
- [ ] 場景 5: 驗證合約回應中包含完整的 `full_text` 欄位

**優先級**: P0

---

### 需求 4: 合約關鍵資訊提取

**業務目標**:
合約文件通常需要提取特定的關鍵資訊(如合約雙方、金額、日期等),以便後續的業務處理或資料分析。智能提取這些資訊可大幅提升使用者效率。

**驗收標準** (使用 EARS 格式):
1. 系統 SHALL 從合約中提取合約編號(支援多種格式如「合約字第 XXX 號」、「Contract No: XXX」)
2. 系統 SHALL 識別並提取甲方與乙方的名稱(公司名稱或個人姓名)
3. 系統 SHALL 提取合約中的金額數字,並識別貨幣單位(新台幣、美元等)
4. 系統 SHALL 提取簽訂日期與生效日期,支援民國年與西元年格式
5. WHEN 合約包含多個金額時,系統 SHALL 提取所有金額並標註其對應的條款
6. WHEN 合約包含地址時,系統 SHALL 提取並結構化地址資訊(縣市、區域、街道)
7. WHERE 啟用 LLM 輔助提取時,系統 SHALL 使用視覺修正來提高關鍵欄位的準確率
8. IF 關鍵資訊提取信心度低於 70%,THEN 系統 SHALL 標記該欄位為「需人工確認」

**測試場景**:
- [ ] 場景 1: 使用標準格式合約,驗證所有關鍵欄位均能正確提取
- [ ] 場景 2: 測試包含多個金額的合約,驗證所有金額都被提取並正確對應條款
- [ ] 場景 3: 測試民國年日期格式(如「中華民國 115 年 3 月 26 日」),驗證正確轉換
- [ ] 場景 4: 測試信心度低於 70% 的情況,驗證系統正確標記「需人工確認」
- [ ] 場景 5: 使用 LLM 輔助提取,對比純 OCR 與 LLM 輔助的準確率差異

**優先級**: P1

### 需求 5: 可擴展的架構設計

**業務目標**:
系統應該易於擴展,使未來新增其他文件類型(如發票、身分證、報表等)時,不需要大幅修改核心架構,僅需添加新的處理器與回應格式定義。

**驗收標準** (使用 EARS 格式):
1. 系統 SHALL 採用策略模式(Strategy Pattern)設計,每種文件類型對應一個獨立的處理策略
2. 系統 SHALL 提供統一的文件處理器介面,所有文件類型處理器必須實作此介面
3. 系統 SHALL 使用配置檔管理支援的文件類型清單,新增類型時只需修改配置
4. 系統 SHALL 將回應格式定義與處理邏輯分離,使用結構化的 Schema 定義
5. WHEN 新增文件類型時,系統 SHALL 無需修改 API 路由層與控制器層的程式碼
6. 系統 SHALL 提供文件類型處理器的抽象基類,定義共用的預處理與後處理流程
7. 系統 SHALL 支援文件類型處理器的熱載入,無需重啟服務即可新增類型

**測試場景**:
- [ ] 場景 1: 在配置檔中新增一個測試文件類型,驗證系統能自動識別
- [ ] 場景 2: 實作一個最小化的文件類型處理器,驗證能順利整合到系統中
- [ ] 場景 3: 檢視程式碼結構,確認文件類型處理邏輯與 API 層解耦
- [ ] 場景 4: 測試熱載入機制,驗證無需重啟即可啟用新文件類型

**優先級**: P1

## 非功能性需求

### 效能需求
1. 系統 SHALL 在 30 秒內完成單頁合約文件的 OCR 處理(不含 LLM 修正)
2. 系統 SHALL 在 2 分鐘內完成 10 頁合約文件的完整處理(含 LLM 修正)
3. 文件類型判斷與路由 SHALL 在 100ms 內完成,不顯著增加整體處理時間
4. 系統 SHALL 支援並行處理多個不同類型的文件請求,不互相阻塞

### 可用性需求
1. 系統 SHALL 維持現有謄本功能的 100% 向後兼容性,不破壞現有用戶的使用體驗
2. 錯誤訊息 SHALL 使用繁體中文,明確指出錯誤原因與修正建議
3. 系統 SHALL 在日誌中記錄文件類型、處理時間、提取成功率等關鍵指標

### 安全性需求
1. 系統 SHALL 驗證上傳文件的類型與大小,防止惡意文件上傳
2. 系統 SHALL 對提取的個人資料(如姓名、地址)進行脫敏處理,僅在需要時顯示
3. 系統 SHALL 限制 API 請求頻率,防止資源濫用

### 可維護性需求
1. 系統 SHALL 提供詳細的單元測試,覆蓋率達 80% 以上
2. 文件類型處理器 SHALL 具備獨立性,一個類型的問題不影響其他類型
3. 系統 SHALL 提供監控端點,報告各文件類型的處理成功率與平均時間

## 限制與假設

### 技術限制
1. 合約文件的頁數上限為 50 頁,超過則需分批處理
2. 結構化資訊提取依賴文件的標準格式,非標準格式可能提取失敗
3. LLM 視覺修正功能需要 OpenAI API 金鑰,未配置時將跳過
4. 系統僅支援繁體中文合約,簡體中文或英文合約需要額外配置

### 業務假設
1. 假設使用者知道上傳文件的類型,不需要系統自動分類
2. 假設 `data/contracts/` 中的 11 份範例涵蓋常見的合約格式
3. 假設合約中的關鍵資訊以文字形式存在,而非純圖像(如掃描簽名)
4. 假設使用者接受結構化資訊提取可能不完美,需要人工覆核

## 相依性

### 上游依賴
1. 現有的 `document-ocr-enhancement` 功能必須穩定運作
2. `backend/app/lib/ocr_enhanced/` 套件需要支援策略模式擴展
3. `data/contracts/` 目錄必須包含至少 5 份可用的測試合約
4. OpenAI API(可選)用於 LLM 輔助的關鍵資訊提取

### 下游影響
1. 前端測試界面 `OcrTestView.vue` 需要更新以支援文件類型選擇
2. API 文檔系統需要重新生成,反映新的參數與回應格式
3. 未來的文件管理系統(`/api/v1/documents/`) 可能需要整合此功能
